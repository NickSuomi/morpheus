# Agent Instructions

Always read project guidance before working:

- `docs/product/PRD.md`
- `CONTEXT.md`
- `ARCHITECTURE.md`
- relevant files in `docs/adr/`
- `docs/agents/*.md`

Use `.understand-anything/knowledge-graph.json` as the repo-owned architecture
map after reading the canonical docs above. Start with its `tour`, then
`layers`, then targeted `nodes` and `edges` for package-boundary work.

Use Beads for local task tracking. Do not create markdown task lists as tracker substitutes.
`.beads/` is local/private project state and must stay ignored by public git.
Do not commit Beads exports, Beads history, private target signoff evidence, or
target-specific task evidence to Morpheus.

Keep public product vocabulary as Morpheus. Use Sandcastle only for the low-level adapter around `@ai-hero/sandcastle`.

Do not commit private target repository names, private hostnames, private issue/MR URLs, local auth paths, token values, or target-specific evidence to Morpheus. Use anonymized fixture/public target names in docs, tests, and commits. Keep Beads local/private, and keep private target signoff evidence outside Morpheus git.

History cleanup for leaked private target data is destructive and requires
explicit operator approval. See `docs/agents/private-data-history-cleanup.md`.
Run `pnpm scan:private-data` before publishing release or signoff changes. Set
`MORPHEUS_FORBIDDEN_PRIVATE_PATTERNS` to the current operator-approved private
target names, hostnames, and path fragments before running the gate.

## Commits

Use `.gitmessage` as the commit template.

Commit messages should include:

- why this change exists
- what changed
- how it was verified
- Beads issue refs when applicable

Do not use flat one-line commits for project work. If committing from CLI, pass
multiple `-m` blocks or use the editor so the message follows `.gitmessage`.
Tiny typo-only commits may omit the body, but planning, architecture, and task
tracking commits should include the structured sections.

## Beads Issue Tracker

Use Beads (`bd`) for durable local task tracking in repositories that include it. Use the `beads` skill at `.agents/skills/beads/SKILL.md` (project install) or `~/.agents/skills/beads/SKILL.md` (global install) for Beads workflow guidance, then use the `bd` CLI for issue operations.

### Quick Reference

```bash
bd ready                # Find available work
bd show <id>            # View issue details
bd update <id> --claim  # Claim work
bd close <id>           # Complete work
bd prime                # Refresh Beads context
```

### Rules

- Use `bd` for all local task tracking; do not create markdown TODO lists.
- Run `bd prime` when Beads context is missing or stale.
- Keep persistent project memory in local Beads via `bd remember`; do not create ad hoc memory files.
- Keep `.beads/` ignored and untracked. Publish only scrubbed, intentional task summaries in docs or commits.
